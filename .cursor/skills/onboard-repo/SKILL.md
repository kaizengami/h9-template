# onboard-repo

Universal entry point for "we just got handed a repo, where do we
start." Runs a deterministic bash probe (`scripts/probe.sh`), then
synthesizes an `ONBOARDING.md` that the rest of the team can read in
60 seconds.

This is the **first thing** to run when the hackathon challenge gives
you an existing repo to fix or extend. Every other skill in the
given-repo flow (`reproduce-bug`, `bisect`, `codemod`,
`issue-triage`) assumes onboarding has already happened.

## When to use

- Phase 1 of the hackathon, after the brief lands and you discover the
  challenge gives you 1-6 existing repos. Run **once per repo**.
- Any time you switch into a worktree of a repo you didn't bootstrap.
- Before invoking `@implementer` against unfamiliar code (the agent
  will refuse without an `ONBOARDING.md`).

## How to invoke

```text
/onboard .
```

or

```text
/onboard ../some-other-repo
```

## What it does (procedure)

1. **Run the bundled probe.** Execute
   `bash .cursor/skills/onboard-repo/scripts/probe.sh "$ARGUMENTS"`.
   The script reads file presence, lockfiles, and config, then prints
   a structured report to stdout.
2. **Read the existing AI config**, if any:
   - `AGENTS.md`, `CLAUDE.md`, `.cursor/`, `.claude/`
   - If found, summarize what conventions the original team set up so
     we don't fight them.
3. **Skim the README** with `Read`. Extract the one-paragraph "what
   does this thing do" — don't paraphrase, quote when possible.
4. **Surface hot files** from the probe output (`git log` frequency).
   These are the highest-leverage files to read next.
5. **Check open issues**, if it's a GitHub repo: `gh issue list
   --limit 10 --json number,title,labels`.
6. **Synthesize ONBOARDING.md** at the repo root using the template
   below. Keep it under 80 lines.
7. **Report back** to the calling session: "ONBOARDING.md written.
   Suggested next step: <one sentence>."

## ONBOARDING.md template

```markdown
# ONBOARDING.md

> Generated by /onboard at <ISO timestamp>. P1 should review and edit.

## What this repo is
<One paragraph, ideally quoted from README. If README is absent, your
best one-paragraph guess based on file structure.>

## Stack at a glance
- **Language(s):** <e.g. TypeScript 5.4, Python 3.12>
- **Framework:** <e.g. Next.js 15 App Router, FastAPI 0.110>
- **Package manager:** <e.g. pnpm, uv>
- **Test runner:** <e.g. vitest, pytest>
- **Lint/format:** <e.g. eslint+prettier, ruff>
- **CI:** <e.g. GitHub Actions in .github/workflows/>

## How to run it
```bash
# Install
<pnpm install OR uv sync OR …>

# Dev / serve
<pnpm dev OR uvicorn app.main:app --reload OR …>

# Test
<pnpm test OR pytest OR …>

# Lint
<pnpm lint OR ruff check . OR …>
```

## Key files (hot zones from git log)
- `path/to/file:1` — <one-line description>
- `path/to/other:1` — <one-line description>
- `path/to/third:1` — <one-line description>

## Existing AI config (if any)
- `AGENTS.md`: <present/absent — one-line summary if present>
- `CLAUDE.md`: <present/absent — one-line summary if present>
- `.cursor/`: <present/absent — list of subdirs>
- `.claude/`: <present/absent — list of subdirs>

## Open issues (if GitHub)
- #N: <title> [labels]
- #N: <title> [labels]

## Open questions for the team
1. <Anything unclear about intent or conventions — leave for human
   decision; don't guess.>
2. ...

## Suggested next step
<One sentence. e.g. "P2 picks up issue #42 (auth bug). Run
`/repro 42` first." Or "Stack is unfamiliar; @architect to
review the build pipeline before we touch anything.">
```

## Rules

1. **Never edit code from this skill.** Onboarding is read-only
   investigation plus writing the ONBOARDING.md output file.
2. **Quote, don't paraphrase the README.** Saves accuracy questions
   later.
3. **If the repo has its own `AGENTS.md` or `CLAUDE.md`, respect it.**
   Don't impose hack9 conventions on an unfamiliar repo. Note the
   conflict in "Open questions" if there's a real tension.
4. **Don't guess at the stack.** If the probe can't identify the test
   runner or package manager, write `unknown — ask the team` rather
   than making something up. Wrong run commands waste 20 minutes.
5. **Cite every file claim with `path:line`.** Especially in "Key
   files" — if you can't cite a line, you didn't really read it.
6. **80-line cap on ONBOARDING.md.** It's a brief, not a tutorial. If
   you need more, link out.

## Failure modes

- **Probe times out (e.g. huge repo).** Kill it after 30 seconds, fall
  back to manual `Glob`/`Grep` for the essentials (lockfile, README,
  package.json or pyproject.toml). Note in ONBOARDING.md that probe
  was incomplete.
- **No git history** (shallow clone or freshly initialized). Skip the
  hot-files section and write "git history unavailable."
- **No README.** Use the top-level directory listing as a stand-in.
  Write "no README found; structure-based guess" in the summary.

## Don't

- Don't mirror the hack9 `.cursor/` setup into the new repo. We're a
  visitor.
- Don't run `pnpm install` or `pip install` from this skill. That's
  the calling session's decision.
- Don't `gh issue triage` or comment on issues. That's
  [issue-triage](../issue-triage/SKILL.md)'s job.
