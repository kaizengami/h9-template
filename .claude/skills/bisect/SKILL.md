# bisect

Wrapper around `git bisect run` for finding the commit that introduced
a regression. Use when the bug is "this used to work and now it
doesn't" — bisect is the most-leveraged investigative tool for that
specific class of bug.

This skill is **manual invocation only** (`disable-model-invocation:
true`) because bisect mutates the working tree (checks out arbitrary
historical commits) and shouldn't auto-fire.

## When to use

- A regression is confirmed (it works on commit X, fails on commit Y).
- The repro command is **deterministic** — same exit code every run.
  Flaky repros will produce wrong bisect results.
- The repo's history between X and Y is at least ~10 commits;
  otherwise just read the diffs manually, faster than bisect.

## When NOT to use

- The repro is non-deterministic (network calls, time, randomness
  without a seed). Bisect will be useless.
- You don't actually know a "good" sha. Bisect needs both endpoints.
- The change spans a rebase/squash-merge that obscured history. Read
  the squash commit's diff directly instead.

## How to invoke

```text
/bisect <good-sha> <bad-sha> <repro-cmd>
```

Examples:

```text
/bisect v1.2.0 HEAD pnpm test auth -- -t "login"
```

```text
/bisect abc123 def456 pytest tests/test_email.py::test_unicode_subject
```

The repro command must:
- Exit 0 when the bug is **absent** (the commit is "good").
- Exit non-zero when the bug is **present** (the commit is "bad").
- Run in under ~60 seconds. Bisect on N commits runs the command
  log2(N) times. 30 commits = ~5 invocations = 5 minutes if each is a
  minute.

## Procedure

1. **Validate inputs.** Three positional arguments required:
   `<good-sha> <bad-sha> <repro-cmd...>`. If any missing, refuse with
   usage text. Confirm both shas resolve via `git rev-parse`.
2. **Confirm the working tree is clean.** Run `git status --porcelain
   | head -1` — if non-empty, refuse with: "working tree dirty —
   commit or stash before bisecting."
3. **Capture the current branch** so we can return to it. Save the
   bisect log path: `BISECT_LOG=/tmp/bisect-$(date +%s).log`.
4. **Generate the bisect step script** at `/tmp/bisect-step.sh`:
   ```bash
   #!/usr/bin/env bash
   # Step script for git bisect run. Generated by /bisect skill.
   set -uo pipefail
   <REPRO_CMD>
   exit $?
   ```
   Mark executable: `chmod +x /tmp/bisect-step.sh`.
5. **Start the bisection:**
   ```bash
   git bisect start "<bad-sha>" "<good-sha>"
   ```
6. **Run it:**
   ```bash
   git bisect run /tmp/bisect-step.sh 2>&1 | tee "$BISECT_LOG"
   ```
   This will iterate, checking out commits and running the step script
   on each. Output is captured for the final report.
7. **Capture the verdict.** When `git bisect` finishes, the offending
   sha is in the bisect log. Extract with:
   ```bash
   bad_commit=$(git bisect log | tail -2 | head -1 | awk '{print $2}')
   ```
   (Or scan for "first bad commit" in the run output.)
8. **Show the diff:**
   ```bash
   git show --stat "$bad_commit"
   git log -1 --format='%H%n%an <%ae>%n%ad%n%n%B' "$bad_commit"
   ```
9. **Reset bisect state:**
   ```bash
   git bisect reset
   git checkout <original branch>
   ```
10. **Report back** with this format:
    ```
    ## Bisect result

    Offending commit: `<sha>`
    Author: <name>
    Date: <date>
    Subject: <commit subject>

    ## Files touched
    <git show --stat output>

    ## Suggested next step
    Read the diff (`git show <sha>`) — the regression is in this
    commit. Most likely root cause: <one-line guess based on the
    files touched>. Now invoke `@bug-hunter` to confirm, or `@implementer`
    if the fix is obvious.

    Bisect log: <BISECT_LOG path>
    ```

## Failure modes

- **Bisect finds no bad commit** (the entire range is "good"). Means
  the bad-sha was wrong or the bug was reverted. Report and stop.
- **Step script errors out (exit 125, "skip this commit")** for many
  commits. Means the build is broken in that range. Report the commits
  that couldn't be tested and stop.
- **User aborts mid-bisect.** Always run `git bisect reset` in your
  cleanup; never leave the repo in a detached HEAD state.

## Rules

1. **Never run bisect on a dirty tree.** Step 2 is non-negotiable.
2. **Never `git bisect reset` without reporting first.** If you reset
   before extracting the verdict, the result is lost.
3. **Single repro command per invocation.** If the bug needs setup
   (e.g. seed the database), put the setup inside the step script via
   inline bash:
   ```text
   /bisect v1 HEAD "set -e; pnpm db:reset; pnpm test auth -- -t login"
   ```
4. **Time-box the operation.** If a single step takes more than 90
   seconds, abort the bisect and tell the user "repro is too slow for
   bisect — recommend manual diff inspection."

## Don't

- Don't bisect across major framework upgrades — the build won't
  compile in older commits, every step skips, bisect finds nothing.
- Don't combine bisect with codemod history rewrites. Codemod commits
  touch hundreds of files and obscure the real change.
- Don't bisect for "performance got slower" without a hard threshold
  in the step script (e.g. fail if test takes > 5 seconds). Without a
  threshold, the step script always exits 0.
